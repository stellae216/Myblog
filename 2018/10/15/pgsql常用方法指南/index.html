trigger触发器

创建方法upd_timestamp()，并创建trigger应用到表对应列上

advertiser_updated_trigger: 为trigger名称，advertiser为表名

new, old, tg_op ： 都是trigger中内置的属性，分别代表新实例，旧实例，事件类型， tp_op => 'INSERT'/'UPDATE'/'DELETE'

before: 指触发器执行时间，还有after ，注意: before：(insert、update)可以对new进行修改，after不能对new进行修改，两者都不能修改old数据。

update: 指触发器的事件，还有insert / update / delete，同时使用多个事件，使用or连接

on table: 针对表触发，还可以针对列触发of field1, field2 on table

for each row : 指触发器类型，还有for each statement

  • 触发器的 for each row 属性是可选的，如果选中，当操作修改时每行调用一次
  • 选中for each statement，不管修改了多少行，每个语句标记的触发器执行一次

when <condition>: 触发条件

物化视图

PostgreSQL中的物化视图像视图一样使用了规则系统，但是以一种类表的形式保留了结果。物化视图不能直接被更新，并且用于创建物化视图的查询的存储方式和视图查询的存储方式完全相同。

使用物化视图能够有效解决复杂的视图查询慢的问题，缺点是数据不实时，需要一个定时任务定期刷新数据。

  • 创建物化视图
  • 刷新物化视图数据

1、全量刷新：默认为全量刷新，将会使用更少的资源并且完成地更迅速，但是物化视图将保留在不可扫描的状态，直至刷新完成。

2、增量刷新：参数CONCURRENTLY(version >=9.4)，将当前视图表中的数据和query中的数据做一个join操作，然后才将差量做填充。增量刷新会比全量刷新更耗费资源并且更慢，在物化视图上有至少一个UNIQUE索引才能使用，但是不会阻止物化视图被扫描

  • concurrently 参数为增量刷新数据
  • 定期刷新可以参考文章
  • 删除物化视图

Tip: DROP MATERIALIZED VIEW 删除已存在的物化视图。要执行该命令必须是该物化视图的所有者。

注意！使用物化视图生成的表字段默认都是允许为空，并且不能做修改。如果有特殊需求，可以在物化视图的基础上，新建其复制表，这样就可以正常添加索引和设置字段约束，如下所示：

分区表

当数据达到千万级甚至更高时，数据查询就会比较耗时了，这时我们可以利用表分区来提升查询性能。

创建分区有两种方式：(1)、DeclarativePartitioning定义分区表；(2)、使用表继承(Inheritance)方式定义分区表。我们使用第一种即可满足大部分业务场景。一般如果可以直接查询子表，查询效率会比查询主表要好。

  • 使用表继承(Inheritance)方式定义分区表，见文章：PostgreSQL-12分区表详解 \ postgres表分区

  • pgsql同样支持多级分区，见文章：pgsql多级分区

  1. 创建分区主表(partition by range)，指定范围的数据落入指定的分区(前闭后开)，pg10引入。
  • 在 PostgreSQL 10 中，分区上的索引需要基于各个分区手动创建，而不能基于分区的父表创建索引。PostgreSQL 11 可以基于分区表创建索引。分区表上的索引并不会创建一个物理上的索引，而是为每个分区上的索引创建一个模板。也就是说，如果在分区表上创建了一个索引，PostgreSQL 自动为每个分区创建具有相同属性的索引。
  • FROM (0) TO (10) -> 取0-9
